## Workflow Instructions

Determine whether you are working on an **Issue** or a **Merge Request** based on the URL above and the assignment context.

### Issue Workflow (working on a new feature or bug fix)

1. **Read the full context**: Use `gitlab-connect issue view [id]` to read the issue description and all comments. Understand the complete scope of work before starting.

2. **Create a feature branch**: Create a descriptive branch name based on the issue (e.g., `feature/issue-[id]-short-description` or `fix/issue-[id]-short-description`).

3. **Implement the work**: Make the necessary code changes to address the issue requirements. As you work:
   - Commit and push code regularly with descriptive commit messages
   - Post summary comments on the issue via `gitlab-connect issue comment [id] --file /tmp/comment.md` to keep stakeholders informed of progress

4. **Test thoroughly**: Ensure all feature work is fully tested. Add unit and regression tests to the existing test suite where appropriate.

5. **Update documentation**: If the feature affects user-facing behavior or developer workflows, update existing project documentation as necessary.

6. **Open a merge request**: Once the feature is complete and all tests pass, create a merge request targeting the main branch. Use `gitlab-connect mr create --title "..." --file /tmp/mr_body.md` with a clear description of changes.

### Merge Request Workflow (addressing code review feedback)

1. **Read the full context**: Use `gitlab-connect mr view [id]` to read the MR description and all review comments. Understand all requested changes before starting.

2. **Address all feedback**: Implement the requested changes from code reviewers. Consider all comments for guidance on the expected approach.

3. **Commit and push**: As changes are implemented, commit and push to the existing branch.

4. **Post update comments**: Use `gitlab-connect mr comment [id] --file /tmp/comment.md` to notify reviewers that feedback has been addressed and the MR is ready for re-review.

## Guidelines

- Use `/tmp/` for any temporary files needed for commit messages, comments, or other content. If your built-in file-writing tool cannot write to `/tmp/`, use a shell command instead (e.g., `printf '%s' 'content' > /tmp/comment.md`). Clean up temporary files after use.
- If any other GitLab issues or merge requests are referenced, review them with `gitlab-connect issue view [id]` or `gitlab-connect mr view [id]`.
- Run the project test suite before finalizing work to ensure no regressions.
- Do NOT close issues or merge MRs automatically - leave them open for human review.
- Do NOT emit `/assign` commands in your comments to avoid triggering recursive work assignments.

## Completion Criteria

Once all workflow steps are complete (code changes committed, tests passing, MR opened or review feedback addressed), post a final summary comment on the issue or MR and then **exit immediately**. Do not wait for further instructions or user input. Your task is finished when the workflow above is complete.
